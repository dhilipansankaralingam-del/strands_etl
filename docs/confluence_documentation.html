Strands ETL Orchestrator Framework

Version 1.1.0 Production Python 3.x

Overview: The Strands ETL Orchestrator (etl_orchestrator.py) is a config-driven, end-to-end data pipeline framework that automates S3 stale folder detection, AWS Glue job triggering, post-load data quality validations via Athena, cross-table comparison checks, summary analytics with chart generation, audit logging, and HTML email dashboard reporting. It supports three run modes: full_pipeline, validation_only, and summary.

3. Run Modes

Usage: python etl_orchestrator.py --config s3://bucket/path/orchestrator_config.json [--summary-config s3://bucket/summary_config.json]
Run ModeDescriptionPhases ExecutedWhen to Use
full_pipeline Complete end-to-end orchestration S3 Stale Detection → Glue Trigger → Validations → Comparisons → Audit → Email Scheduled daily runs; new data landing from source systems
validation_only Skip stale detection & Glue, run validations directly Validations → Comparisons → Audit → Email Re-run validations after manual loads; spot-check data quality
summary Parallel analytics queries with chart generation Summary Queries + Charts → (optionally) Validations → Audit → Email Generate dashboards & reports; weekly/monthly summaries
Fallback Logic: If run_mode=summary but no summary queries are defined, the orchestrator falls back to validation_only. If no validations are defined either, it falls back to full_pipeline.

4. Phase 1: S3 Stale Folder Detection

The orchestrator monitors configured S3 prefixes for stale data subfolders. Source systems (MACP, MRD) drop CSV files into timestamped subfolders under each prefix. The staleness check determines whether data needs processing.

4.1 Staleness Algorithm

ConditionResult StatusAction
Subfolder count > stale_subfolder_count_threshold (default: 3)StaleProcess ALL subfolders via Glue
Any subfolder age > stale_subfolder_hours_threshold (default: 3h)StaleProcess only stale subfolders via Glue
No subfolders & last update > no_subfolder_hours_threshold (default: 5h)Check with SourceLog warning, skip Glue
No objects found under prefixCheck with SourceLog warning, skip Glue
Everything within thresholdsOKNo action needed

5. Phase 2: Glue Job Triggering

For each source classified as Stale, the orchestrator triggers an AWS Glue job to process the stale CSV subfolders into Iceberg tables.

5.1 Glue Job Flow

StepActionDetail
1Build argumentsPass --stale_folders (comma-separated), --json_file_name, --bucket_name, --archive_s3_path, --source_bucket, --orchestrator_run_id
2Start Glue jobglue_client().start_job_run(JobName, Arguments, WorkerType, NumberOfWorkers, Timeout)
3Poll for completionCheck status every 30 seconds until terminal state: SUCCEEDED, FAILED, STOPPED, TIMEOUT, ERROR
4Record resultLog job_name, job_run_id, status (PASS/FAIL), duration_seconds

5.2 trigger_glue.py (Glue Script)

The Glue job script (trigger_glue.py) processes CSV files from S3 into Iceberg tables:

StepAction
1Read config JSON from S3 (schema, database, table name)
2If stale_folders provided: only read CSVs from those subfolders
3Lowercase all column names, cast to target dtypes
4Add processing indicators (hourly_processed_ind, daily_processed_ind) + timestamps (PST/UTC)
5INSERT INTO glue_catalog.{database}.{table_name} (Iceberg)
6Archive processed files, delete stale subfolders

6. Phase 3: Post-Load Validations (Athena)

After Glue loads complete, the orchestrator runs Athena queries to validate data quality across all layers. Validations can run in parallel (up to 5 threads) or sequentially.

6.1 Single-Query Validation Check Types

Check TypePurposeCondition ExampleFail Behavior
row_count Verify rows were loaded cnt > 0 FAIL if count is zero; abort_on_failure: true
no_rows Ensure no unwanted rows (e.g., duplicates) row_count == 0 FAIL if rows found (duplicates detected)
threshold Numeric condition check (nulls, freshness, etc.) null_cnt == 0, hours_since < 24 FAIL if threshold breached
rolling_average Anomaly detection vs N-day baseline |variance_pct| <= tolerance FAIL/WARN if variance exceeds tolerance %
source_reconciliation Compare source vs target counts per table |src - tgt| < deviation_threshold FAIL if deviations found; lists all deviating rows

6.2 Configured Validation Queries

#NameCheck TypeDatabaseAbort on FailureDescription
1landing_row_count_checkrow_countlanding_dbYesVerify rows loaded into landing table (cnt > 0)
2duplicate_checkno_rowslanding_dbYesCheck for duplicate primary keys (row_count == 0)
3null_key_checkthresholdlanding_dbYesVerify no NULL primary keys (null_cnt == 0)
4freshness_checkthresholdlanding_dbNoVerify data within 24h (hours_since < 24)

6.3 Athena Query Execution with Retry

# Retry logic (1 retry on failure)
result = _run_athena_query_once(query, database, output_location, workgroup)
if result["state"] != "SUCCEEDED" and ATHENA_RETRY_ATTEMPTS > 0:
    time.sleep(3)
    retry_result = _run_athena_query_once(...)
    retry_result["cost_usd"] += result["cost_usd"]  # accumulate cost from both attempts
    return retry_result
return result

# Cost tracking per query
data_scanned_bytes = stats.get("DataScannedInBytes", 0)
cost_usd = (data_scanned_bytes / (1024**4)) * 5.00  # $5.00 per TB

6.4 Rolling Average Check Detail

The rolling_average check type compares today's count against an N-day rolling average baseline:

FieldSourceDescription
today_cntQuery result columnCurrent day's record count
baseline_cntQuery result column (avg_Nday_cnt)N-day rolling average baseline
variance_pctCalculated or from query|today - baseline| / baseline * 100
toleranceConfig (default: 25%)Maximum allowed variance percentage
window_start/endQuery resultBaseline calculation window
Dashboard Rendering: Rolling average checks get dedicated visual cards in the email dashboard, showing today vs baseline values with a variance gauge bar (green/orange/red based on proximity to tolerance).

7. Cross-Query Comparison Checks

Comparison checks run two separate Athena queries (Query A + Query B) and compare their results using configurable rules.

7.1 Comparison Rules

RuleLogicUse Case
matchA == BExact row count match between landing and staging
percentage_within|A-B|/max(|A|,|B|) * 100 <= tolerance%Row counts within 5% between landing and curated
difference_within|A - B| <= toleranceAbsolute count difference check
a_not_zeroA != 0Verify non-empty landing load
a_greater_than_bA > BLanding should have more rows than filtered curated
data_diffRow-level delta comparisonFind missing, extra, and changed records between source & target
customUser-provided Python expressionComplex business logic comparisons

7.3 Data Diff (Row-Level Reconciliation)

The data_diff rule performs row-level comparison using composite keys:

Config FieldTypeDescription
key_columnslist[str]Columns forming the composite key (e.g., ["membership_number"])
compare_columnslist[str]Non-key columns to check for value mismatches (e.g., ["member_name", "plan_code"])
case_sensitiveboolCase-sensitive key matching (default: true)
max_diff_rowsintMaximum rows reported per category (default: 100)
ignore_duplicatesboolDe-duplicate keys before diff (default: false)

Detection Categories:

  • Only in A (Missing from Target): Keys present in source but not target
  • Only in B (Extra in Target): Keys present in target but not source
  • Value Mismatches: Same key, different column values (with per-column diff)

9.3 Useful Audit Queries

-- Latest run summary
SELECT run_id, event_type, status, detail
FROM   audit_db.etl_orchestrator_audit
WHERE  run_date = cast(current_date AS varchar)
ORDER  BY event_timestamp;

-- Failure history (last 7 days)
SELECT run_date, run_id, event_type, source_label, validation_name, status, detail
FROM   audit_db.etl_orchestrator_audit
WHERE  run_date >= cast(date_add('day', -7, current_date) AS varchar)
  AND  status IN ('FAIL', 'ABORTED', 'Stale', 'Check with Source')
ORDER  BY run_date DESC, event_timestamp;

-- Cost breakdown by event type
SELECT event_type, SUM(CAST(cost_usd AS DOUBLE)) as total_cost
FROM   audit_db.etl_orchestrator_audit
WHERE  run_date >= cast(date_add('day', -30, current_date) AS varchar)
GROUP  BY event_type;

12. Validation Queries & SQL Patterns

12.1 Row Count Check

SELECT COUNT(*) as cnt
FROM landing_db.{table_name}
WHERE load_date = current_date
-- Condition: cnt > 0  |  Check type: row_count  |  Abort: YES

12.2 Duplicate Detection

SELECT primary_key, COUNT(*) as dup_cnt
FROM landing_db.{table_name}
GROUP BY primary_key
HAVING COUNT(*) > 1
LIMIT 10
-- Condition: row_count == 0  |  Check type: no_rows  |  Abort: YES

12.3 Null Key Check

SELECT COUNT(*) as null_cnt
FROM landing_db.{table_name}
WHERE primary_key IS NULL
-- Condition: null_cnt == 0  |  Check type: threshold  |  Abort: YES

12.4 Freshness Check

SELECT MAX(load_timestamp) as latest_ts,
       date_diff('hour', MAX(load_timestamp), current_timestamp) as hours_since
FROM landing_db.{table_name}
-- Condition: hours_since < 24  |  Check type: threshold  |  Abort: NO

12.5 Landing vs Source Row Count Match

-- Query A (Landing):
SELECT COUNT(*) AS cnt FROM landing_db.macp_address_tb
WHERE cast(load_timestamp as date) = current_date

-- Query B (Source Staging):
SELECT COUNT(*) AS cnt FROM staging_db.macp_address_tb_stg
WHERE cast(load_date as date) = current_date

-- Rule: match  |  Compare: cnt  |  Abort: YES

12.6 Percentage Tolerance Check

-- Query A (Landing):
SELECT COUNT(*) AS cnt FROM landing_db.mrd_mbr_info
WHERE cast(load_timestamp as date) = current_date

-- Query B (Curated):
SELECT COUNT(*) AS cnt FROM curated_db.mrd_mbr_info
WHERE cast(update_date as date) = current_date

-- Rule: percentage_within  |  Tolerance: 5%  |  Abort: NO

12.7 Data Diff (Row-Level Delta)

-- Query A (Source Delta):
SELECT membership_number, member_name, plan_code
FROM landing_db.mbr_delta WHERE cast(load_timestamp as date) = current_date

-- Query B (Target Delta):
SELECT membership_number, member_name, plan_code
FROM curated_db.mbr_delta WHERE cast(update_date as date) = current_date

-- Rule: data_diff  |  Key: membership_number  |  Compare: member_name, plan_code
-- Detects: missing rows, extra rows, value mismatches

12.8 Summary: Daily Load Counts

14. Troubleshooting Guide

IssuePossible CauseResolution
Glue job timeoutLarge CSV files, slow S3 readsIncrease timeout_minutes in source config; upgrade worker type
Athena query slow / expensiveFull table scan without partitionsAdd partition filters (WHERE load_date = current_date); review table format
Athena cost spikeUnoptimized queries scanning full tablesEnsure partition pruning in all queries; convert to columnar format
Stale folder not detectedSubfolder age below thresholdLower stale_subfolder_hours_threshold; check source system delivery schedule
MSCK REPAIR failureMalformed partition pathVerify S3 path format run_date=YYYY-MM-DD/; run manually in Athena
Email not receivedSMTP connectivity / authVerify SMTP host/port reachable; check firewall rules for port 25
Validation shows NOT_EXECUTEDAthena query failed after retryCheck failure_reason in audit; verify database/table names; check IAM permissions
All comparisons ABORTEDPrior validation with abort_on_failure failedFix the critical validation first; remaining checks auto-abort
Rolling average shows WARNToday's count deviates > tolerance%Review source data volume; adjust tolerance if seasonal pattern
Charts missing in emailmatplotlib not installedInstall matplotlib; bar charts use HTML fallback, pie/trend require matplotlib
Data diff finds mismatchesTiming difference between source and target loadsEnsure both queries use same date filter; verify transformation logic
Debug Tip: All events are logged to the audit table with run_id for traceability. Query the audit table filtered by run_date and status IN ('FAIL', 'NOT_EXECUTED', 'ABORTED') to quickly find issues.
